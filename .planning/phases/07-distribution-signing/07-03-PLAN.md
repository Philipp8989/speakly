<task type="checkpoint:human-action">
  <name>Task 1: Configure GitHub secrets</name>
  <what-needed>
    Before the release CI can run, all signing secrets must exist in the GitHub repository settings.
    Claude cannot access GitHub repository settings — this step requires the repository owner.
  </what-needed>
  <how-to-complete>
    1. Open: GitHub repo → Settings → Secrets and variables → Actions
    2. Add each secret listed in `.planning/phases/07-distribution-signing/SECRETS.md`

    Required secrets (9 total):
    - APPLE_CERTIFICATE (base64 .p12)
    - APPLE_CERTIFICATE_PASSWORD
    - KEYCHAIN_PASSWORD (any random string)
    - APPLE_SIGNING_IDENTITY (e.g. "Developer ID Application: Your Name (TEAMID)")
    - APPLE_ID (your Apple Developer email)
    - APPLE_TEAM_ID (10-char string from developer.apple.com)
    - APPLE_PASSWORD (app-specific password from appleid.apple.com)
    - TAURI_SIGNING_PRIVATE_KEY (from: pnpm tauri signer generate)
    - TAURI_SIGNING_PRIVATE_KEY_PASSWORD

    If you do NOT have an Apple Developer account yet, the macOS build will succeed but produce an unsigned .dmg. Gatekeeper will warn on install. You can still verify SC-3 (universal binary) and SC-4 (CI pipeline runs) without a Developer account. SC-1 (no Gatekeeper warning) requires the account.

    Type "secrets configured" when done, or "skip signing" if you want to verify the pipeline without signing secrets.
  </how-to-complete>
  <resume-signal>Type "secrets configured" or "skip signing"</resume-signal>
</task>

<task type="auto">
  <name>Task 2: Push tag and verify GitHub Actions run completes</name>
  <files>(none — git operations only)</files>
  <read_first>
    - .github/workflows/release.yml — confirm it exists before tagging
  </read_first>
  <action>
    Commit any uncommitted changes from Plans 01 and 02, then push the release tag:

    ```bash
    # Stage all changes from Plans 01 and 02
    git add src-tauri/tauri.conf.json .github/workflows/release.yml

    # Commit if there are staged changes
    git diff --cached --quiet || git commit -m "chore(release): add CI workflow and finalize distribution config"

    # Push the main branch first
    git push origin main

    # Create and push the v0.1.0 tag
    git tag v0.1.0
    git push origin v0.1.0
    ```

    Then monitor the workflow run using gh CLI:
    ```bash
    gh run list --workflow=release.yml --limit=3
    ```

    Wait for the run to appear (it starts within 30 seconds of the tag push). Then watch it:
    ```bash
    gh run watch
    ```

    The run will take approximately 15-25 minutes (macOS cross-compilation is slow). Do NOT interrupt.

    After completion, verify the GitHub Release was created:
    ```bash
    gh release view v0.1.0
    ```

    Expected output shows .dmg and .msi assets attached to the release.

    If the workflow fails:
    1. `gh run view --log-failed` — examine the failure
    2. Fix the issue (most common: wrong APPLE_SIGNING_IDENTITY format, base64 encoding of cert)
    3. Delete and re-push the tag: `git tag -d v0.1.0 && git push origin :v0.1.0 && git tag v0.1.0 && git push origin v0.1.0`
  </action>
  <verify>
    <automated>gh release view v0.1.0 --json assets --jq '.assets[].name' 2>/dev/null | grep -E '\.(dmg|msi)' && echo "Release artifacts present"</automated>
  </verify>
  <done>GitHub Release v0.1.0 exists with at least one .dmg and one .msi asset. gh release view confirms both files are downloadable.</done>
</task>
